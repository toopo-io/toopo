# ADR 0012: Database — Neon Postgres + Drizzle ORM

Date: 2026-05-15
Status: Superseded by ADR-0017

## Context

Phase 4 needs a relational database for auth (user, session, account,
verification) and, in later phases, generated assets, jobs, and
billing. The constraints:

- **Serverless-friendly**: TOOPO's deploy target is Vercel/Render-style
  ephemeral compute. Long-lived pools cost money and reconnect on cold
  start.
- **Branchable**: preview environments per PR shorten the feedback
  loop on schema changes.
- **TypeScript-first ORM**: schemas should be the source of truth for
  TS types, mirroring ADR-0006 (Zod as source of truth on the wire).
- **Transactional**: Better Auth issues transactions; the driver must
  support them.

## Decision

- **Neon Postgres** as the managed database. HTTP + WebSocket
  serverless drivers, generous free tier, branchable in seconds,
  built-in connection pooler.
- **Drizzle ORM 0.45** as the data-access layer.
- **`drizzle-orm/neon-serverless` (WebSocket driver)** as the runtime
  connection mode. Supports transactions (required by Better Auth)
  and pooled connections through a single `DATABASE_URL`. The HTTP
  driver is faster for one-shot reads but doesn't support
  transactions; `pg` is fine but requires the Neon pooler URL
  separately.
- **`@toopo/db`** (build-distributed shared package per ADR-0004)
  owns `createDb({ databaseUrl })` and the schema. The factory is
  lazy — `Pool` is constructed without opening a socket until the
  first query.

### Schema lives in `.ts` modules (ADR-0010 category 1)

`packages/db/src/schema/auth.ts` defines the four Better Auth canonical
tables (`user`, `session`, `account`, `verification`) plus our
`user.deleted_at` extension. The barrel `schema/index.ts` re-exports
them so `drizzleAdapter` consumes them as `* as schema`.

### Migrations live outside `src/` (ADR-0010 category 2)

`packages/db/drizzle/migrations/` holds SQL migration files written by
`drizzle-kit generate`. Drizzle Kit reads `packages/db/drizzle.config.ts`
which points at `./src/schema/index.ts` for the source-of-truth and
`./drizzle/migrations` for the output. Migrations are **explicit only**
(`pnpm db:migrate`) — never auto-run on app boot. Per ADR-0008's
fail-fast philosophy, silent migration on boot is a footgun in
production.

## Why not @better-auth/cli for schema generation

Q6 in the Phase 4 plan asked us to use `@better-auth/cli` to scaffold
the Drizzle schema. We tried; it didn't fit.

At Phase 4 time:

- `better-auth` latest stable: **1.6.11**.
- `@better-auth/cli` latest stable: **1.4.21**, with **exact** peer
  pin on `better-auth@1.4.21` (no caret) and `drizzle-orm@^0.41.0`.
- No CLI version yet released for `better-auth@1.6.x`. The CLI
  consistently lags the runtime library.

Running CLI@1.4.21 against runtime@1.6.11 means the schema is
generated by an *older* `better-auth` than the one our API runs.
That removes the conformance guarantee Q6 wanted — a CLI lagging the
runtime can emit a stale schema and we'd never know.

**Decision**: hand-write the schema in `packages/db/src/schema/auth.ts`
using the official Better Auth Drizzle documentation as reference.
Pin `better-auth` with tilde (`~1.6.11`) so patch upgrades flow but
minor bumps require manual schema review per the checklist below.

### Drift mitigation (documentation, not detection)

The hand-write decision means schema drift is a maintenance hazard
with no automated detector. Three documentation surfaces converge on
the upgrade checklist below:

1. `auth.ts` header doc-block cites the upstream URL, lists custom
   extensions, and points back here.
2. `packages/db/README.md` explains the hand-write rationale and
   points back here for the procedure.
3. The "Updating better-auth checklist" below is the canonical
   procedure.

Automated detection (CI diff against `@better-auth/cli` output, or
migration applied to preview DB) remains an open gap — see
"Operational gap (not closed)" below.

### Updating better-auth checklist

Before bumping `better-auth` (especially across minor versions):

1. **Read the release notes** at
   https://github.com/better-auth/better-auth/releases for breaking
   changes to schema fields, defaults, or auth flows between the
   pinned and target versions.
2. **Diff the canonical schema**: open
   https://better-auth.com/docs/adapters/drizzle for the target
   version and verify column names, types, nullability, and indexes
   match `packages/db/src/schema/auth.ts`. Fields that have churned
   historically: `session.token`, `account.*Token*`, and the
   `verification` table shape.
3. **Re-apply custom extensions** on top of the new canonical shape.
   Currently only `user.deletedAt` (RGPD — see ADR-0013).
4. **Generate the Drizzle migration** with `pnpm db:generate` from
   `packages/db/` and review the emitted SQL by hand before
   committing.
5. **Exercise the live flow** locally against a real Postgres:
   signup → verify-email → sign-in → reset-password. The unit tests
   do not run a real Better Auth flow against the schema (see
   "Operational gap (not closed)" below).
6. **If a preview database is available**, apply the migration
   there before merging.

**Exit condition**: if `@better-auth/cli` ever ships with reliable
peer ranges that include our pinned runtime, reconsider using it on
the next major upgrade.

## Operational gap (not closed)

CI does not run migrations against a preview database. Drizzle Kit's
ergonomics for per-PR Neon branches + automated `db:migrate` need a
follow-up ADR. The current state is: developers run migrations
locally; production runs them as a deploy step. Phase X (deploy
hardening) closes this gap.

## Alternatives considered

- **Prisma**: rich tooling, but heavier runtime, slower cold-start on
  serverless, and historically less stable on transactions vs Drizzle.
- **Supabase**: bundles auth + DB. We separated those concerns
  (Better Auth + Neon) for portability and to avoid tight vendor
  coupling.
- **`drizzle-orm/neon-http`**: rejected — no transaction support.
- **`drizzle-orm/node-postgres` + `pg`**: works, but means managing
  the Neon pooler URL separately. `neon-serverless` is purpose-built.

## Known dependency advisory

As of 2026-05-18, `pnpm audit` flags a moderate vulnerability
(GHSA-67mh-4wv8-2f99) in `esbuild ≤0.24.2`, reached transitively via
`drizzle-kit > @esbuild-kit/esm-loader > @esbuild-kit/core-utils >
esbuild`.

Assessment:

- Dev-only: `drizzle-kit` is not used at production runtime.
- The bug affects the esbuild dev server, not the bundled artifacts.
- TOOPO production builds are unaffected.

Action:

- Monitor `drizzle-kit` releases for an updated `@esbuild-kit/*`
  dependency.
- If the advisory escalates or impacts CI, apply a pnpm `overrides`
  pin to force a patched esbuild version.

This note exists to prevent re-investigating the same alert at every
`pnpm audit` run.
